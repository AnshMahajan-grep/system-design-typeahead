# Search Typeahead System

A search suggestion / autocomplete system like the one search engines and
e-commerce sites use. It suggests popular queries as you type, records search
submissions, and ranks suggestions by a blend of **all-time popularity** and
**recent activity**.

**Stack:** Node.js + Express · PostgreSQL · Redis (×3 nodes) · React (Vite) ·
Docker Compose.

---

## Features (maps to the assignment rubric)

| Requirement | Where it lives |
|---|---|
| Typeahead suggestions (top 10, prefix match, sorted by count) | `backend/src/services/suggestionService.js`, `routes/suggest.js` |
| Search UI with debounced suggestions + keyboard nav | `frontend/src/App.jsx`, `useDebounce.js` |
| Dummy `POST /search` returning `{ message: "Searched" }` | `backend/src/routes/search.js` |
| Query-count data store (durable) | PostgreSQL `queries` table — `backend/src/db/schema.sql` |
| Caching for low latency (cache → DB fallback, TTL expiry) | `backend/src/cache/cacheCluster.js`, `suggestionService.js` |
| **Distributed cache via consistent hashing** | `backend/src/cache/consistentHash.js` |
| Trending searches (recency-aware ranking + decay) | `backend/src/services/trendingService.js` |
| **Batch writes** (buffer + aggregate + bulk flush) | `backend/src/services/batchWriter.js` |
| Dataset ≥ 100k queries with counts | `backend/scripts/seed.js` (generates 120k) |
| Performance report (p95 latency, hit rate, write reduction) | `GET /metrics`, `backend/scripts/benchmark.js` |

📄 **Full project report — architecture, dataset, API docs, design choices &
trade-offs, performance numbers, and screenshots — is in
[`REPORT.md`](REPORT.md).**

## Quick start (Docker — the intended way)

> Prerequisite: **Docker Desktop** installed and running.

```bash
# from the project root
docker compose up --build
```

This starts 6 containers: `postgres`, `redis-0`, `redis-1`, `redis-2`,
`backend`, `frontend`. On first boot the backend automatically:
1. creates the Postgres schema, and
2. seeds **120,000** generated queries (idempotent — skipped on later boots).

Then open:

- **UI:**        http://localhost:5173
- **API:**       http://localhost:8080
- **Metrics:**   http://localhost:8080/metrics

To stop: `Ctrl+C`, then `docker compose down` (add `-v` to also wipe the seeded
database volume).

### First boot takes a minute
Seeding 120k rows runs once. Watch the backend logs (`docker compose logs -f
backend`) for `Seed complete`. The API answers `/health` as soon as it's up.

## Running without Docker (local dev, optional)

You still need Postgres and the three Redis instances. Easiest is to run just
those in Docker and the apps on your host:

```bash
# 1. infra only
docker compose up postgres redis-0 redis-1 redis-2

# 2. backend
cd backend
cp .env.example .env          # points at localhost ports 5432 / 6390-6392
npm install
npm run seed                  # load the dataset
npm start

# 3. frontend (new terminal)
cd frontend
npm install
npm run dev                   # http://localhost:5173
```

## Dataset

- **Source:** generated by `backend/scripts/seed.js` — a realistic, head-heavy
  (Zipf-like) distribution of e-commerce/tech search queries composed from real
  vocabulary (`iphone 15 pro`, `java tutorial for beginners`, …).
- **Size:** 120,000 unique queries (> the 100k minimum).
- **Why generated:** the assignment permits any text dataset and explicitly
  allows deriving counts. Generation is fully reproducible, needs no Kaggle
  credentials or network, and is easy to explain in the viva.
- **Use a real CSV instead:** put `query,count` rows in
  `backend/scripts/dataset.csv` and set `SEED_FROM_CSV=1`.

## Try it from the command line

```bash
# suggestions (recency-aware by default)
curl "http://localhost:8080/suggest?q=iph"
# basic popularity-only ranking
curl "http://localhost:8080/suggest?q=iph&mode=basic"

# submit a search (records it; returns the dummy response)
curl -X POST http://localhost:8080/search -H "Content-Type: application/json" -d '{"query":"iphone 15 pro"}'

# which cache node owns a prefix? hit or miss?
curl "http://localhost:8080/cache/debug?prefix=iph"
# how prefixes spread across the 3 nodes
curl "http://localhost:8080/cache/debug/distribution"

# trending right now
curl "http://localhost:8080/trending"

# performance metrics
curl "http://localhost:8080/metrics"
```

To watch recency in action: `POST /search` the same query ~20 times, then
compare `GET /suggest?q=<prefix>&mode=basic` vs `&mode=recency` — the freshly
searched query jumps up in recency mode and appears in `/trending`.
